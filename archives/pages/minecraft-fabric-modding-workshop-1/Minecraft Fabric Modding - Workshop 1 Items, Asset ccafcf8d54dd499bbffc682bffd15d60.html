Items

There are three major components to adding an item:

  1. Declaring & registering the item
  1. Adding a model & texture
  1. Adding a recipe or other way to obtain it

None of these are particularly hard, though it does take some precision and a good workflow to avoid errors. Let’s first declare the item.

Declarations

We are going to declare a couple of items at the same time so you can compare and contrast the process of making them. The first will be something called “Brilliant Alloy” and the second will be “Gilded Netherite Ingot”, both of which will be used in later workshops as well. First, highlight your init package and right-click. Select New > Java Class and name this class something like ‘ItemInit’. This is where we will declare all of our items. Import our main class as “import net.workshop.main;”. Then, we’ll make two methods, neither of which need parameters. I will name mine registerItems() and init(). Then, within init(), call registerItems(). So far, you should have something like:

package net.workshop.init;

import net.workshop.main;

public class ItemInit {


    public static void registerItems() {

    }

    public static void init() {
        ItemInit.registerItems();
    }
}

Now we can start declaring the items. Inside the public class, but before registerItems(), we’ll add a line for each new item. The example given on the wiki goes as follows, replacing anything with <> with a name of our choice (also remove the <> symbols):

public static final Item <ITEM_NAME> = 
	new Item(new FabricItemSettings().group(ItemGroup.MISC));

Traditionally, the ITEM_NAME is written in all caps, and uses underscores ( _ ) any time you want to use a space. Therefore, I replaced <ITEM_NAME> with BRILLIANT_ALLOY in the first declaration and GILDED_NETHERITE_INGOT in the second. The ending .group tag adds this item to the MISC tab of the creative menu. By default, it will put it at the very end of the list. We can change this if we like, but for now, it’s not necessary. As well, for the Gilded Netherite Ingot, I am going to add an extra tag to its declaration. After the .group(ItemGroup.MISC), I will put .rarity(Rarity.RARE) so that this item has a cooler name color. This is what items like golden apples and potions use to show that they are valuable. You also probably got red letters throughout these. This is because we didn’t import the libraries we’re referencing yet. If you hover over a red section, it should have a small popup with “Import Class” under the error. Click on that and the error will go away given that the code now recognizes the imported libraries.

  • As well, there are other times that multiple options for an import can come up, and sometimes these are parts of JetBrains libraries and not Minecraft’s. Whenever you have multiple options for an import, usually you should go with the one that references Minecraft
  • Weirdly, there are two different places where rarity is mentioned. When you import that part, pick the one with ‘util’ as part of its name/path

So far, we should done the following:

  • Created two new declarations, one for a brilliant alloy and one for gilded netherite ingot
  • added a .rarity tag to gilded netherite ingot’s declaration
  • Have 4 imports in our ItemInit class

Next, we’ll add to the body of registerItems(). This will be the registry so the item actually gets added to the game. The base code looks like this:

Registry.register(Registry.ITEM, new Identifier(<namespace>, "<item_name>"), <ITEM_NAME>);

However, we’ll need to be more careful this time. First, <namespace> was already declared. Our namespace is the main.MOD_NAME inside the main class, the global variable we declared. Therefore, replace the whole of <namespace> with main.MOD_NAME.

Next, we need to be consistent with our format. Notice that <ITEM_NAME> appears again. Therefore, we should copy exactly what we named our item before, as in BRILLIANT_ALLOY and GILDED_NETHERITE_INGOT. This goes for both items; their corresponding registry line should have the exact same all-caps name.

Next, for <item_name> inside the parentheses, it’s the same thing as <ITEM_NAME> but lowercase. Funny enough, we should follow that exact format. Copy whatever you replaced <ITEM_NAME> with and place it inside the quotation marks, replacing the whole of <item_name>. However, it will be all lowercase instead of uppercase, so instead of BRILLIANT_ALLOY, it will be brilliant_alloy in the quotations. This is once again a precedent set by Mojang themselves for how they declare and register stuff, so we’ll be consistent with them.

  • Notes on this part: what the namespace is telling us is where to find the assets we will create later. Our namespace is ‘workshop’, so later we’ll work in a resource folder under assets called ‘workshop’
  • As well, the lowercase <item_name> is the name of an asset file we’re making later. When we make that file, we’ll make sure that its name is the exact same as the lowercase <item_name>we placed here

Lastly, we need to go back to our main class. Within the onInitialize() method, we’ll call our init() method in ItemInit. So now it looks like:

@Override
public void onInitialize() {
	ItemInit.init();
}

Here’s what we just did:

  • Registered BRILLIANT_ALLOY and GILDED_NETHERITE_INGOT in registerItems()
  • Added ItemInit.init(); to the main class’s onInitialize() method

With this, our items have been registered! You can runClient and check the MISC tab of the Creative menu to see that they’re both there.

However, you can immediately tell they’re very unfinished. They don’t have textures, and their names are their internal reference names. Let’s solve both of these by adding to the resources section of the mod.

Asset Files

Asset files are resources that add new things to our declared items. This is where we keep track of new sounds, models, textures, language files for naming items in-game, and much more that we can touch on in later workshops.

Modeling & Texturing

Under resources, we have assets/workshop. Right-click workshop and select New > Directory and call in ‘models’. Then, make another directory called ‘item’ inside ‘models’. This is so that we can tell the difference between an item model and a block model (which we’ll be doing far in the future). Now, inside ‘item’, make a New > File. Name this file exactly whatever you used before in <item_name>, the lowercase version. As well, be sure to add .json at the end of the name, as that will tell IntelliJ that this is a json file. Now that we have our model file, we’ll add the below code to it, replacing anything with <> again (don’t forget to remove the <> symbols too):

{
  "parent": "item/generated",
  "textures": {
    "layer0": "<namespace>:item/<item_name>"
  }
}

What this is doing is telling the mod what kind of model this is (generated is a basic item, held in the hand) and where the texture file is. The namespace references the name of the directory over models (workshop), and <item_name> tells it the name of the texture file to find and use for this item (which is right after this). Do this for both items. In this case, it will be in a directory we haven’t made yet, so let’s make it. Right-click workshop again and create a new directory called ‘textures’, and inside of that create a directory called ‘item’. Now we’ll need a texture png. Item textures are typically 16x16 pixels in size with a transparent background. To make your own, you’ll need some kind of photo editing program that supports transparency. Windows’s Paint does not. A free one is Gimp, though it seems a bit cumbersome to learn. Personally, I use PaintTool SAI, but that requires a license. For this workshop, I can provide all necessary textures required. For our two items here, we will add two png files named exactly as <item_name>.png for both, just like what we listed in their model files. Here they are:

They’re very small, but you can click on the files on the left and save them to add them to the project. Place them into the textures/item directory and we’ll be good.

Data Files

Data files tell us about the functionality of our modded stuff. It is here that we create new advancements, recipes, and tags that give our stuff additional uses in the game.

Recipes

Now we’ll need to add some data to our mod so we can obtain these items. First, we’ll do a shaped recipe for the brilliant alloy, then two recipes for the gilded netherite ingot, one for a furnace and one for a blast furnace. Select the resources folder and make a new directory titled “data”. Then, within that, create a directory with the same name as your namespace (mine is workshop). Then create another directory inside that title “recipes”. Lastly, make a new file title “<item_name>.json”. This file will be structured like so:

{
  "type": "minecraft:crafting_shaped",
  "pattern": [
    "123",
    "123",
    "123"
  ],
  "key": {
    "1": {"item": "<namespace>:<item_name1>"},
    "2": {"item": "<namespace>:<item_name2>"},
    "3": {"item": "<namespace>:<item_name3>"}
  },
  "result": {
    "item": "<namespace>:<item_name>",
    "count": 1
  }
}

Now it gets a bit tricky. Notice the three lines of “123” after each other. That actually mimics the 3x3 grid of a crafting table, so wherever you place a symbol is where an ingredient will go. Next, we have the “key” section. This tells the recipe what each symbol in the recipe means. For instance, wherever a 1 appears could be a diamond, so for key “1”, we list “item”: “minecraft:diamond” (Minecraft is the namespace for all items normally in the game). The “result” says what we get out of the recipe, and how many of the items. For my recipe for brilliant alloy, I went with 1 netherite ingot, 4 gold ingots, and 4 copper ingots. As well, here’s a link to the full list of items, and you can click on them to find their internal reference names on their pages

  • https://minecraft.fandom.com/wiki/Item

Now for the recipes for gilded netherite ingots. The structure for a furnace recipe is:

{
  "type": "minecraft:smelting",
  "ingredient": {
    "item": "<namespace>:<item_name>"
  },
  "result": "<namespace>:<item_name>",
  "experience": <int>,
  "cookingtime": <int>
}

Notice the change in the “type” at the top of the file. “item” refers to what we put in, “result” is what we get. “experience” is the number of experience we get when we take the item out of the furnace. Typically, an item of this level still only gives around 3-5 xp. “cookingtime” is how long it’ll take to cook. The standard for a furnace is 200. The blast furnace recipe is very similar, except the “type” is now “minecraft:blasting” and the “cookingtime” is cut in half (so 100 typically).

After all of this, the first section of your first items are (mostly) done! You’ve checked off every box at the start of the page.

Language Files

You might have noticed that in one above photo, my Brilliant Alloy item actually had a name, and not item.workshop.brilliant_alloy. That’s because I used a language file to rename it. We can add a language file to our mod that can give in-game names to all of our items. That way, their name in-game is not their internal reference name. To do this, we’ll need a new directory under resources > assets > workshop. Title this directory ‘lang’, which Fabric automatically knows is a language folder. Then add a New > File called en_us.json, which will act as an extension to Minecraft’s English US language file. Within this file, we will first reference the internal name of our items in quotations, then use a string in quotations to say what we want the item to display as in-game. This looks like:

{
	"<type>.<namespace>.<item_name>": "<in-game name>"
}

In this case, type is item, and you can quickly look up your namespace and item_name as needed. The second string can be anything you’d like and will be exactly how Minecraft names the item in-game. The name you see here is a copy of the string I put in.

Recipe Advancements

This section is dedicated to adding our recipes to the recipe book once we have the proper materials. Under resources > data > workshop, create a new directory called “advancements”. In that, we’ll need two JSON files, one for the recipe for Brilliant Alloy and one for both recipes of Gilded Netherite Ingot. I named my files “get_recipe_brilliant_alloy” and “get_recipe_gilded_netherite”. The names aren’t very important, the mod will load them anyhow, but they’re for you to keep track of their contents. Let’s do the brilliant alloy first. In its advancement file, use the following structure: 

{
  "parent": "minecraft:recipes/root",
  "rewards": {
    "recipes": [
      "<namespace>:<recipe_name>"
    ]
  },
  "criteria": {
    "<criteria_name>": {
      "trigger": "minecraft:inventory_changed",
      "conditions": {
        "items": [
          {
            "items": ["<namescape>:<item_name>"]
          }
        ]
      }
    }
  }
}

The first recipe_name reference is to whatever item’s recipe you want. In this case, brilliant alloy’s, so it’s namespace is ‘workshop’ again, and the recipe_name is the name of json file we made that has the recipe. The second item is a reference to whatever item has to be in your inventory to trigger you learning the recipe, and last is criteria_name, which in this context can be whatever you like. I went with “get_netherite_ingot”. Similarly, the second item for me was a netherite ingot, so once I first have one in my inventory, I’ll learn the recipe to brilliant alloy. For gilded netherite ingots, the code will look very similar, but I’ll replace the netherite ingot reference with brilliant alloy. As well, the recipe listings will instead list the namespaces and file names of both JSON recipe files we made for gilded netherite ingots.
