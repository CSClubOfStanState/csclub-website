Declaring Our Block

Our special directional blockstates force us to make a superclass for our block. The reason for this is it will be in our superclass that we grab the direction the player was facing, and it will then apply that direction when determining the block’s direction. Fortunately, there isn’t much extra we have to do for the class. Go to java > net > workshop and make a new package called block. Inside that package, make a new class called BlockGildedNetherite. This class will extend Block, and use the following super:

public BlockGildedNetherite(Settings settings) {
        super(settings);
        setDefaultState(this.stateManager.getDefaultState().with(Properties.FACING, Direction.NORTH));
    }

Now we need to grab the directional methods This will require two @Overrides to the appendProperties method and the getPlacementState method. These look like this:

@Override
    protected void appendProperties(StateManager.Builder<Block, BlockState> stateManager) {
        stateManager.add(Properties.FACING);
    }

    @Override
    public BlockState getPlacementState(ItemPlacementContext ctx) {
        return this.getDefaultState().with(Properties.FACING, ctx.getPlayerFacing().getOpposite());
    }

The appendProperties() adds the FACING property to the block’s metadata, meaning that it it has a hidden value to say what direction the block is facing. This is very important for making directional blocks. Then, getPlacementState() uses the .getPlayerFacing() method to see what way the player was facing when they placed this block, and then gets its opposite value and applies that to this block’s FACING property. The reason for getting the opposite value is because when we place the block, we want it to “face us”, as opposed to facing the same direction as us. So far, we should have done the following:

  • Made a package under workshop called block
  • Made the class BlockGildedNetherite inside of package block
    • declared its super
    • set it’s default state to FACING NORTH
    • appended the property of FACING
    • used getPlacementState() to get player direction

With these, there is now directionality that we can apply to our block later, and we can start our declarations and registries. Our declarations are going to look quite similar, that much has not changed. However, we need to stay organized and keep our format. Go to java > net > workshop > init and make a new class titled BlockInit. We are going to set up this class in the same format we did ItemInit, in that we’ll have declaration first, then our register method, then the init method to run the other methods in the class. You can copy this format:

public class BlockInit {

    // Declarations

    public static void registerBlocks() {

		    // Registry
    }

    public static void init() {
        BlockInit.registerBlocks();
    }
}

A declaration for a basic item goes as follows in Fabric:

public static final <superclass> <BLOCK_NAME> = new <superclass>(FabricBlockSettings.of(Material.<MATERIAL>).strength(<float hardness>F, <float resistance>F).requiresTool());

The .requiresTool() at the end is not necessary, but for our purposes we’ll need it. To finish that logic, we’ll use tags later to say what tool is needed for this block. Now Let’s go through these items

  • the <superclass> is typically Block, but we just declared our own special superclass, so we’ll use our special one instead
  • <BLOCK_NAME> is formatted just like <ITEM_NAME> from before. I went with GILDED_NETHERITE_BLOCK
  • <MATERIAL> determines the group of blocks this will go in with. Considering the netherite and gold used to make it, I went with METAL
  • In .strength(), we have two float values to place. The first is for hardness, which controls how slow the block is to break. The second refers to blast resistance, as in how strong of an explosion do you need to break this block. For both of these, I went with 4.0F, as we can easily change them later to better fit our want

Now we can register our block. Go into the registerBlocks() method and add these two lines:

Registry.register(Registry.BLOCK, new Identifier(main.MOD_NAME, "<block_name>"), <BLOCK_NAME>);
Registry.register(Registry.ITEM, new Identifier(main.MOD_NAME, "<block_name>"), new BlockItem(<BLOCK_NAME>, new FabricItemSettings().group(ItemGroup.<GROUP>)));

Notice that we’re actually declaring an item in the second line. That’s because we need to register both the block and its corresponding item. All of the sections that need to be replaced are things that you have more or less seen before, so you can go ahead and finish these statements off. As well, I’ll be adding a .fireproof() and .rarity() tag to the ITEM registry to be consistent with our other gilded items. Now we should have accomplished these:

  • Made a new class under init called BlockInit
  • Declared new block GILDED_NETHERITE_BLOCK
    • Added .requiresTool() clause to declaration
  • Registered new block GILDED_NETHERITE_BLOCK
  • Registered new block-item GILDED_NETHERITE_BLOCK
    • Added .fireproof() and .rarity() to the block-item

When we check in-game, we have our new block! Untextured and unmodeled, but we can fix that right now.

Modeling Blockstates

We need to tell the game what directions this block will have, and how to orient the block in this situations. To do this, we’ll need an asset file. Go to resources > assets > workshop and make a new directory called blockstates. All blocks have a blockstate, but most just have one, their default. For us, we’ll need to declare each direction the player could be facing. Make a JSON file in this directory called what we placed in the quotations of the registry, the lowercase <block_name>. In it follow this template:

{
  "variants": {
    "facing=north": { "model": "<namespace>:block/<block_name>" },
    "facing=east":  { "model": "<namespace>:block/<block_name>", "y": 90 },
    "facing=south": { "model": "<namespace>:block/<block_name>", "y": 180 },
    "facing=west":  { "model": "<namespace>:block/<block_name>", "y": 270 }
  }
}

We’ll replace the <namespace> and <block_name> with our namespace and block name of course. It will be the same in each of these listings. Very cool, that FACING property we added to the superclass before is the value that’s being passed to this file, and that’s how it determines which way to face the block. We can also see that we have taken care all all four cardinal directions, and that the “y” value increases by 90 for each face other than north, which is our default. Think of it like the y-axis of the game is on a pole and you turn it 90 degrees to rotate the block. That’s how the math works out. With this file, the game will apply our directional states, but now we need to add the textures. I actually also want to make a block that has different textures on each face, which will require some extra work in the model file. Go to assets > workshop > models and make a new directory called block. In that, make a JSON file titled after our <block_name>. We can copy this format for our multi-textured block:

{
  "parent": "block/cube",
  "textures": {
    "particle": "workshop:block/<block_name>",
    "up": "workshop:block/<block_name>",
    "down": "workshop:block/<block_name>",
    "north": "workshop:block/<block_name>",
    "south": "workshop:block/<block_name>",
    "east": "workshop:block/<block_name>",
    "west": "workshop:block/<block_name>"
  }
}

The “particle” texture is what determines the breaking particles appear as we break this block. We’ll use the texture “workshop:block/gilded_netherite_block_nw”. The reason for the nw is because that stands for northwest, the direction that the diagonal line in the texture is facing. More on that later, it’ll make more sense when you see the texture. Then are the textures for each face of the block. The direction refers to the cardinal direction that face of the block is closest towards. For example, the “north” face is the one that “looks” at the north. As for textures, we have the two variants: “workshop:block/gilded_netherite_block_nw” and “workshop:block/gilded_netherite_block_ne”. In order to pick the correct one for each, we’ll say that the “top” is ne. From there, if we picture the cube, the diagonal lines should always connect to another. Here’s a helpful visual aid:

For each of these textures, rotate this picture so that the text on that side is correctly oriented for you. Then, place the texture into the references as you see it for that side. If you textured the block properly, these diagonal lines on the textures will pleasingly connect. The last thing to do is give the item a model, which is easy. Go to models > item and make the file gilded_netherite_block.json. In it, we just need this:

{
  "parent": "workshop:block/gilded_netherite_block"
}

Now the item will model itself just like the block, but item-sized. So with these three files, we should have done the following:

  • Made a new directory under resources > assets > workshop called blockstates
    • Made the file “gilded_netherite_block.json” in that directory
    • Replaced each “<namespace>:block/<block_name>” with “workshop:block/gilded_netherite_block”
  • Made a new directory under resources > assets >workshop > models called block
    • Referenced one of our two textures for each listing in this model file
  • Added a models/item JSON file called gilded_netherite_block.json and had it reference its parent model file in models/block

Lastly, we can add the textures. Go to assets > workshop > textures and make a new directory called ‘block’. In that directory, place these two files:

Data Files

Now we can finish up our block’s functions by adding it to a loot table. A loot table is a file that takes care of any and all drops. When you open a chest in a dungeon, it rolls on its loot table to determine what’s inside the chest. Every block has a loot table, the most common being that you roll with 100% probability to get 1 of that block. This is what we’ll do here. Go to resources > data > workshop and make a new directory called loot_tables. In that directory, make another called blocks. Then, inside of that directory, make a JSON file called <block_name>.json. The typical format for this file goes as follows:

{
  "type": "minecraft:block",
  "pools": [
    {
      "rolls": 1,
      "entries": [
        {
          "type": "minecraft:item",
          "name": "<namespace>:<blockitem_name>"
        }
      ],
      "conditions": [
        {
          "condition": "minecraft:survives_explosion"
        }
      ]
    }
  ]
}

You can see that there is a “conditions” sections that must be met in order to roll on this table. If the block survives an explosion, which is taken care of by the resistance float we declared for it earlier, and it was broken by a valid tool, it will roll on this table. We only roll once, and the only entry is our blockItem. All we have to do is replace the “name” with our namespace and blockItem’s reference name. Now we need to use tags to complete our .requiresTool() clause in the declaration. Go the data > minecraft > tags, which we made a couple workshops ago, and add a new directory called blocks. Under this directory, we’ll add a file called needs_diamond_tool.json. Following the same format as our tags before, we will add our new block to the tag, as:

{
  "replace": false,
  "values": [
    "<namespace>:<block_name>"
  ]
}

Then, still under blocks, make a new directory called mineable. In that directory, make a JSON file titled pickaxe.json. Copy the same contents of the previous tag into here. Now our new block should be mineable by a pickaxe and require at least a diamond pickaxe to drop its item.

Recipes

We can add recipes for the block. First, we should make one for the block, which would be 9 gilded netherite ingots filling the entire crafting table area. Then, we’ll need a recipe to change the block back, which is a shapeless recipe that uses one block and returns 9 ingots.

Recipe Advancements

We can add our new block to our recipe advancement for gilded netherite ingots. As well, we can add a new one for when we have one of these blocks. The recipe for the block will go in with the recipe advancement we’ve made already, and then we’ll make a new one that gives us the recipe for turning the block back into ingots. This will work off of the same inventory_change trigger we’ve been using.

Tags

We can make a new tag using this block so piglins like it. For this, we’ll need to make a directory under tags called “blocks”, since this is not an item. In it, we’ll make a new JSON file called “piglin_loved.json”, the block mirror of the item tag. In this tag, we’ll format it just like the item tags we’ve made, and it’s only entry will be our gilded netherite block. We can also add our new block to the list of acceptable blocks to power beacons with. This is another block tag called “beacon_base_blocks.json”, and we just have to add this block to the entry (do NOT set ‘replace’ to TRUE).
