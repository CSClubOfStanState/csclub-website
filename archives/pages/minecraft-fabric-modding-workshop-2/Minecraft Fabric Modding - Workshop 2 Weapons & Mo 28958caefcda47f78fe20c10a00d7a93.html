Tool Tiers

Tool tiers are what determine many of the attributes of tools and weapons in the game, such as damage, mining speed, durability, and enchantability. They exist in a superclass, a Java class “above” others that use it to determine functions and details about themselves. You can think of it as a superclass being an “outline” for objects and other classes that use it, which then fill out the details. Emerald Tier To start, we’ll place our new tiers in a new package, for the organization. Go to java > net > workshop and make a new package called ‘item’. I like to name item classes with their type in front, then their material. I will make a class called TierEmeraldItem, so alphabetically all the tiers we make are together. There’s a lot that goes inside of this class, so best we just get into it. Here’s the outline, once again replacing anything with <> symbols, including the <>:

package net.workshop.item;

public class TierEmeraldItem implements ToolMaterial {

    public static final TierEmeraldItem INSTANCE = new TierEmeraldItem();

    @Override
    public int getDurability() {
        return <int>;
    }

    @Override
    public float getMiningSpeedMultiplier() {
        return <float>F;
    }

    @Override
    public float getAttackDamage() {
        return <float>F;
    }

    @Override
    public int getMiningLevel() {
        return <int>;
    }

    @Override
    public int getEnchantability() {
        return <int>;
    }

    @Override
    public Ingredient getRepairIngredient() {
        return Ingredient.ofItems(<item>);
    }
}

Be sure to import libraries as we go to get rid of those red errors. To discuss all of this, follow this link to get an idea of how the base game's tiers are structured so you can play around with the numbers as we go through each of these. The beginning class declaration says that we are creating a new tool class that can then be easily reused for when we make our actual tools later, hence “implements ToolMaterial”. The second line lets us apply those attributes to an item we declare as a TierEmeraldItem. This, unfortunately, needs to be the same as the class name, so we can’t shorten it. Kinda wordy, but it’s a good organization, which makes up for it in my opinion. Here’s a chart explaining the different between simply initializing a superclass vs. implementing a superclass (and also extending a superclass, which we’ll do next workshop).

Next are all the attributes of those tools. I would like emerald items to be “glass cannons”, in that they’re very effective, but have very low durability, so I’ll put my numbers for each attribute. Feel free to play around with the numbers and see how they differ from mine, though.

  • getDurability sets the tool durability. The number correlated directly to the number of uses the tool has. I went with 35
  • getMiningSpeedMultiplier sets the mining speed for breaking valid blocks with such a tool. This multiplier is based on your hand's mining speed. I went with 15.0F
    • DO NOT delete the F symbol on these numbers, they declare those numbers to be float-point numbers. If you remove them, the code will bug out, since it thinks it’s an integer but tries to apply them to floats
  • getAttackDamage sets the damage bonus that all tools in this class receive. This is separate from attack bonuses from tools (like how swords and axes have higher base damage). I went with 5.0F
  • getMiningLevel sets how tough of blocks tools in this class can break. This is mainly for pickaxes; a higher level (up to 4) means the tool can break tougher blocks like obsidian and ancient debris. I went with 3
  • getEnchantability sets the probabilities for getting stronger enchantments on tools of this class. Higher numbers mean better chances for stronger enchantments. I went with 30
  • getRepairIngredient sets the item type needed to repair this tool class at an anvil. This requires you to know the item ID and registry location used by the code, not what appears in-game. For instance, a diamond is Items.DIAMOND, so we just select emerald instead. Gilded Netherite Tier We’ll also need to make a tier for gilded netherite tools. This format will look very similar, but I’ll be sure to include the important changes. Make a new class inside of the ‘item’ package called TierGildedNetheriteItem. The body will look identical, change numbers however we see fit, of course, except the public static final variable will swap the name out for this class:

public static final TierGildedNetheriteItem INSTANCE = new TierGildedNetheriteItem();

and the ingredient for getRepairIngredient will have to reference our own modded item. To do this, we will preface the reference with the class we declared it in (ItemInit) and then its all-caps ITEM_NAME. So together, it is ItemInit.GILDED_NETHERITE_INGOT. This means that, rather than import Minecraft’s items, we’ll import our own net.workshop.init.ItemInit;. Change up numbers however you like for all other attributes. I’d like these tools to be slightly stronger than Netherite ones, so my numbers were, in order: 2048, 10.0F, 4.5F, 4, 20. Again, you can play with these however you’d like.

  • Made superclass TierEmeraldItem and completed each attribute
    • Float values have F next to number
    • repair ingredient is Item.EMERALD
  • Made superclass TierGildedNetheriteItem and completed each attribute
    • Float values have F next to them
    • repair ingredient is ItemInit.GILDED_NETHERITE_INGOT

Again, be sure to clear those imports. You can see that it references some of the stuff we just declared, like attackDamage and material. When we actually declare the Gilded Netherite Pickaxe, it will use this superclass to place the numbers we give it and use them to build the item in the game. With all classes completed, now we can declare our new tools!

Declaring Tools & Weapons

Now we need to move back to our ‘ItemInit’ class and add the lines to first declare then register our new weapons, like what we did last time. Of course, these are not basic items, so their declarations will include more detail. Firstly, it’s good practice to separate our items with some indents and commas so we can better organize our different declarations. I put the items we made last time first, then broke off for the tools, then weapons. Something like this:

// Items
public static final Item BRILLIANT_ALLOY= 
	new Item(new FabricItem Settings().group(ItemGroup.MISC));
public static final Item GILDED_NETHERITE_INGOT= 
	new Item(new FabricItemSettings().group(ItemGroup.MISC).rarity(Rarity.RARE));

// Weapons
<sword declarations>

public void registerItems() {

Of course we’ll fill in those declarations with actual code. Tools and weapons both follow this format:

public static <superclass> <ITEM_NAME> = 
	new <superclass>(<tier>, <int baseDamage>, <float swingSpeed>, new Item.Settings().group(ItemGroup.COMBAT));

This is lengthy and annoying to navigate through, so it may serve you well to ENTER around the commas to make a more readable list of attributes. You can use this structure just fine:

public static <superclass> <ITEM_NAME> = 
	new <superclass>(
		<tier>, 
		<int baseDamage>, 
		<float swingSpeed>, 
		new Item.Settings().group(ItemGroup.<group>));

Much more readable. Now let’s discuss each piece.

  • The <superclass> will call the tool class we want to use for it. In this case, the superclass is SwordItem
  • Then we give it an <ITEM_NAME>, just like last time. I went for GILDED_NETHERITE_SWORD, and will use similar formatting for the other item
  • The <tier> will call an instance (like a single-use) of the tiers we made before. To do this, the first attribute in the parentheses for a gilded netherite item will be TierGildedNetheriteItem.INSTANCE. Use this same format for the emerald item
  • For <int baseDamage>, we’ll put in an integer. Sword have a base damage of 3, so we’ll follow precedent on this
    • How this damage works is that Minecraft will take the base damage here and add the damage bonus from the material class we made. Those two together will give us the total damage of the item
  • The same goes for <float swingSpeed>. Sword’s use -2.4F
    • The value we are determining here will be measured in full-powered swings per second. What Minecraft does is take the float value here and do 4.0 - <swingSpeed>, and that result is the swings per second
  • Lastly, for <group>, weapons are placed into COMBAT

We should now have 2 declarations: one for Gilded Netherite Sword, and one for Emerald Sword.

  • Separated Items from Weapons with line breaks and comments
  • Declared GILDED_NETHERITE_SWORD
    • Uses TierGildedNetheriteItem.INSTANCE
  • Declared EMERALD_SWORD
    • Uses TierEmeraldItem.INSTANCE

Lastly, we need to register our items. I will separate them just like how I did for the declarations. The registry goes similarly as before:

Registry.register(Registry.ITEM, new Identifier(main.MOD_NAME, "<item_name>"), <ITEM_NAME>);

Remember, this points it to the JSON file we’ll put in resources so that it can properly texture and render the item. And at last, you've added a brand new sword to the game! Hit runClient and go into Creative Mode and check the Tools and Combat tabs, and you should see all of our items. Unlike last time, we can see their combat attributes, and when we use them as tools or weapons, they function accordingly.

Setting Up Resources

Similar to what you did in the last workshop, we can create resources for each of our weapons to use. This includes a .json file and a .png file for each item.

Firstly, navigate back to resources > assets > workshop >models > item, and let’s make two new JSON files, named exactly what you placed into the quotations of the registry listing for each item.

  • ex. “emerald_sword” will have a JSON file named emerald_sword.json
  • when you make each file, add .json to the end of each name, so they automatically become JSON files

For the .json files, you can use the following template:

{
  "parent": "item/handheld",
  "textures": {
    "layer0": "workshop:item/<item_name>"
  }
}

Last time, the parent tag was “item/generated”, which made a tiny item held like a random item. However, this time it is “item/handheld”, meaning that the game will scale them up and place them in your hand to look like a tool or weapon.

We’ll replace <item_name> with the same name as the file because the png files that the JSON file points to will be named exactly the same. Again, I have provided textures for these three items:

Do a runClient to make sure their textures actually applied. If not, double-check the names and contents of your JSON files.

Smithing Recipes

For these items, we are going to make recipes involving the smithing table, like how you make Netherite gear. To start, we’ll navigate back to resources > data > workshop > recipes and make a new JSON file for EACH item, named after them again, like how we named their model files. A smithing recipe has the following format: 

{
  "type": "minecraft:smithing",
  "base": {
    "item": "<namespace>:<item_name>"
  },
  "addition": {
    "item": "<namespace>:<item_name>"
  },
  "result": {
    "item": "<namespace>:<item_name>"
  }
}

First, for the base, that’s the item we first start with. In this case, I want to use a diamond sword as the base, so there I will put minecraft:diamond_sword. Then, for the addition, I want to use an emerald, so it’ll be minecraft:emerald. Lastly, the result is what we make with this recipe, in this case, our emerald sword, so it will be workshop:emerald_sword. For the other recipe, I want to use their regular netherite tool variant as the base, then have the addition be our gilded netherite ingot. See if you can figure out its recipe on your own.

Adding Fireproof & Rarity

You can make these items rare by adding the rarity clause to their declarations, like what we did to Gilded Netherite Ingots last time. There is a similar clause for adding the fireproof feature to weapons, which is what netherite items have so lava does not destroy them. What you can do is go to the .group() clause of any of those declarations and add a single . to the end of it, after it’s closing ) symbol. It should pull up a list of auto-complete suggestions you can add to items, and you can explore some of them and see other things you can add. A fireproof one is among them and is a bit obvious as to which one it is. You do not need to add anything inside the parentheses for this clause to work.

Tags

A tag is essentially a list, or a set, of things in the game. Tags are divided by the thing-type, like an item, a block, a sound, or a game event. What tags are useful for is when you want to apply some function or attribute to multiple things at once, so rather than having to list every single item every time you want to use them, you can throw them once into a tag and then reference the tag instead. This applies today because tags are used for determining what can be used for beacon payment to apply status effects and what piglins are attracted to and will pick up. We will add some of our items to the tag “piglin_loved.json”, and our gilded netherite ingot to “beacon_payment_items.json”. Make new directory under resources > data called “minecraft”, then another in that called “tags”, then another called “items”. Make a file called “piglin_loved.json”, which is the same as the real source file Mojang wrote. In this, we first next to say that we are not replacing the original file, only adding to it. Then we’ll list our two new items. It should look something like:

{
  "replace": false,
  "values": [
    "<namespace>:<item_name>",
		"<namespace>:<item_name>",
		...
  ]
}

With the ... meaning it goes on for however many items we will add. We can add one item, or two, or ten, or a hundred. So long as we format it properly. Our namespace for each of our new items will be “workshop”, since we’re adding items that we made. Let’s add brilliant alloy and all of our gilded netherite items to the tag. Notice that we use the lowercase <item_name> to reference it, so be careful about how you input the names of the items. Now make another file called “beacon_payment_items.json”. For this tag, I only want to add one new item, Gilded Netherite Ingot, into the tag. Now our gilded netherite ingots can be used to power beacons. Do a runClient and test to see if both of our tag additions worked.

Despite being able to use Gilded Netherite Ingots in beacons, its texture does not show up on the GUI. That’s because those textures are added to the GUI itself, so if we want Gilded Netherite on there, we need to edit the GUI png file and replace the old one.
