Refactoring Fabric Clone

To get modding, we’ll download the example fabric project the developers made for us, which is a very handy template. In all honesty, any time you want to create a new fabric project, it’s probably easiest to download this template fresh and then rename folders to best suit your project. This is exactly what we’ll do. To start, we’ll go to this link:

https://fabricmc.net/develop/

The top has a link to the Fabric wiki for lots of helpful information on basic and advanced modding instructions, and a link to the Github repository for the example mod. It also includes documentation on the Fabric API for loading mods, Yarn, and the Fabric Loader for those more technically interested in Fabric’s development. It also includes a snippet of code used in an important Gradle file in the project that tells Gradle what version of Yarn and the Loader to use, what version of Minecraft this mod is for, and what version of Fabric to use. All of this comes with the example project, so we don’t need to worry about it, but if you plan on modding for a different version than 1.18.1, you can select your version and then paste that code into your project.

Organization

We should first organize ourselves. I renamed many of my packages and files to something that I prefer, and should you do the same. You will have to be mindful of what changes you made when comparing your code to mine. This is highly recommended so you can get a feel of how to move files and packages around in IntelliJ, though feel free to rename things to something you prefer. Here’s what I did:

  1. Switched format to “Project Files.” Towards the top left, right above your main folder for fabric-example-mod (or whatever you renamed it), there should be a smaller icon that says “Package” by default. This condenses a bunch of folders together for you if they’re empty, which we don’t want. We want to see every folder, so click on that and instead select “Project Files.” Now we will see the entire project’s structure, no shorthanding.
  1. Renamed fabricmc package to ‘workshop’ and removed the example package. The base template has the following structure for our java classes and main file: java/net/fabricmc/example/... which I do not like. I instead moved the mixin package (another name for a java folder) and ExampleMod into fabricmc. You can do this by highlighting and dragging the package and file to the icon for fabricmc. Then I deleted the example package. Make sure to move mixins and the main file BEFORE deleting the example package, else you’ll delete your important files. I then renamed ‘fabricmc’ to ‘workshop’ (right click on the package, select Refactor, then Rename is the first option). IntelliJ may prompt you on if you want to just rename the package or redo other instances where it’s referenced. Say you want it to refactor other instances too. Do Refactor

  1. Renamed ExampleMod main class file to just ‘main’. This is because we will reference the main file multiple times, so having a short name is nice. Plus, main should not show up again in any class or file name, so it’s easy to avoid close-looking names. Do Refactor
  1. Deleted ExampleMixin file. Mixins are a pretty dense concept we will not be using for awhile, so we don’t need to worry about them for now. Hold onto the mixin package, however, as we will eventually use it. Just not now. Delete Anyway
  1. Created a new package in ‘workshop’ called ‘init’. This stands for initialization, as in the place where we’ll declare and register all of our new items. Right-click on the parent package of main (mine is ‘workshop’) and select New, then select Package. Name this folder ‘init’ or something that tells you that this is where we declare and register all our modded stuff.
  1. Renamed ‘modid’ directory in resources to ‘workshop’. This is to be consistent with the changes made both in the java folder and what we’ll do later in the main class. Under resources > assets, there should be a directory (a resource folder) called modid. I renamed that to ‘workshop’, so you should rename it to whatever you renamed fabricmc before, as per step 1 above.
    1. Pro Tip: any time a path is mentioned in the java folder, it uses periods ( . ), and any time a path goes to resources, it uses forward slashes ( / ).
  1. Renamed ‘modid.mixins.json’ to ‘workshop.mixins.json’, and cleaned up its contents. This file lists off all the mixins the mod uses and where to find them. First, I renamed it so the first part was ‘workshop’ instead of the generic ‘modid’. Then, I opened the file and checked its contents. The “package” should point to the directory for the mixin package (mine is net.workshop.mixin). Then, it has an array listing all mixins. That should now be empty since we deleted the one example mixin, so if there’s anything in it, delete it so it just looks like [], like this:
    {
  "required": true,
  "minVersion": "0.8",
  "package": "net.workshop.mixin",
  "compatibilityLevel": "JAVA_16",
  "mixins": [],
  "client": [],
  "injectors": {
    "defaultRequire": 1
  }
}

    If you’ve followed the previous steps, everything else should be the same.

  1. Double-checked fabric.mod.json file. This file is very important. It tells the mod where to find the mixins.json file, our mod icon png, and our main file. We should not need to touch anything else in here, so we will change these three things with caution.
    1. First, find the “icon” listing and ensure its path is the same as the icon.png (mine is assets/workshop/icon.png) in quotation marks.
    1. Next, find the “main” listing under “entrypoints” and ensure its path is the same as the main class (mine is net.workshop.main) in quotation marks.
    1. Last, find the “mixins” listing just under that and double check its name (mine is workshop.mixins.json). This is NOT a path, it looks for this .json file directly in the resources folder.
    "license": "CC0-1.0",
  "icon": "assets/workshop/icon.png",

  "environment": "*",
  "entrypoints": {
    "main": ["net.workshop.main"]
  },
  "mixins": ["workshop.mixins.json"],

Now that we’ve organized all of this, we can set up our main class. Open up your main class and take a look at some of the comments left by the developers. Their warning about using this class directly will be noted, which is why we separated our init’s. First, check our imports at the top. You can delete all of them, as we’ll only need two for now:

import net.fabricmc.api.ModInitializer;
import net.workshop.init.*;

Then, under the ‘public class’ line, we’ll make a global variable that will be the namespace for our mod. The namespace basically is a label for all the items, entities, and anything else we make so we can separate it from Minecraft’s base stuff. The global variable will be:

public static final String MOD_NAME = "workshop";

You MUST put in whatever you renamed the modid directory under resources. This namespace tells the registry where to find every resource of each item we make, so we need to point it to the right place, or else none of our resources will work.

Lastly, you can see the onInitialize() method, which is the main method. This method will run automatically when the mod starts up every time, so anything else we want to run, we’ll put it in here so the main class will boot it up for us. More on this later.
