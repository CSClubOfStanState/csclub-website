Armor Superclass

We’ll need a superclass for our gilded netherite armor that will store all of its attributes. Go to the java > net > workshop > item package and make a new java class titled ArmorGildedNetherite. As part of the public class declaration, we will add “ implements ArmorMaterial” before the { symbol. Then, we will use the following template:

public class ArmorGildedNetherite implements ArmorMaterial {
    private static final int[] BASE_DURABILITY = new int[] {13, 15, 16, 11};
    private static final int[] PROTECTION_VALUES = new int[] {3, 6, 8, 3};

    @Override
    public int getDurability(EquipmentSlot slot) {
        return BASE_DURABILITY[slot.getEntitySlotId()] * <int>;
    }

    @Override
    public int getProtectionAmount(EquipmentSlot slot) {
        return PROTECTION_VALUES[slot.getEntitySlotId()];
    }

    @Override
    public int getEnchantability() {
        return <int>;
    }

    @Override
    public SoundEvent getEquipSound() {
        return SoundEvents.<SOUND>;
    }

    @Override
    public Ingredient getRepairIngredient() {
        return Ingredient.ofItems(<ITEM>);
    }

    @Override
    public String getName() { return "<armor_namespace>"; }

    @Override
    public float getToughness() {
        return <float>F;
    }

    @Override
    public float getKnockbackResistance() {
        return <float>F;
    }
}

The BASE_DURABILITY tells us the durability of each armor piece, starting with the boots. This will be multiplied by a number we determine later to get the actual durability value. These numbers are also precedented by all armor sets, so we will follow said precedent. PROTECTION_VALUES is the amount of armor each piece gives you, starting with the boots. These values are the same for diamond armor and netherite armor, so we will be consistent. Notice that they add up to 20, or 10 full armor pieces on your HUD. Next are the Overrides, the attributes we will be setting. getDurability is the second half of that equation I mentioned before. Each armor piece will take its BASE_DURABILITY, multiply it by this number, and get its actual durability. Netherite uses 37, so I went with 37 as well. getProtectionAmount formalizes the PROTECTION_VALUES into a function that can be easily called. We do not need to change anything here. getEnchantability takes an integer value to determine how likely it is you get better and stronger enchantments. Netherite has 15 and gold has 25, so I went with 20. getEquipSound will play a sound whenever we equip some of this armor. Given that this is a Netherite set as its base, I imported the SoundEvents library and used SoundEvents.ITEM_ARMOR_EQUIP_NETHERITE. There’s lots of other sounds that you could use, so feel free to scroll through and see what other sounds there are. getRepairIngredient determines what item we need to repair this armor. It is gilded netherite, so naturally we’ll want a Gilded Netherite Ingot. For this, we use ItemInit.GILDED_NETHERITE_INGOT. getName gives this armor its own namespace, like a subset. The armor will later use this to find texture files for the model. Traditionally, we just list off the material type, so I went with “gilded_netherite”. getToughness determines the armor toughness of the armor. Toughness is a complicated thing to discuss, there’s a big formula for it here, but we’ll just follow precedent and make it easy for us. Netherite has 3.0 per armor piece, so we’ll insert 3.0F Lastly, getKnockbackResistance tells us the knockback resistance of the armor. What’s very fun is that this doesn’t work. This attribute does nothing. But we can do a workaround later with mixins to make it work in the additional challenges section. Also, for some reason, the code multiplies this value by 10 somewhere, so we need to put 1/10 of whatever value we want here. Netherite has 1.0 per armor piece, so we’ll put 0.1F. Okay, so that was a LOT. But with that, we can declare our armor now.

Declarations & Registry

Return to ItemInit and add a new sections for // Armor in our declarations. First, we need to declare our new armor material. Use this template:

public static final ArmorMaterial <MATERIAL_NAME> = new <armorSuperclass>;

The <armorSuperclass> will be replaced by a call of the superclass we jsut made, in this case ArmorGildedNetherite(). The <MATERIAL_NAME> can be anything we like, so I followed precedent and went with GILDED_NETHERITE_ARMOR. Now we declare our armor piece, using:

public static final Item <ITEM_NAME> =
            new ArmorItem(<MATERIAL_NAME>, EquipmentSlot.<SLOT>, new Item.Settings().group(ItemGroup.COMBAT));

Replace <MATERIAL_NAME> with the one we just wrote in for before, and the <SLOT> can be HEAD, CHEST, LEGS, or FEET. Lastly is our <ITEM_NAME>, to which I wrote GILDED_NETHERITE_HELMET. You can also add a .fireproof() clause to this so it doesn’t burn in lava, since it’s netherite. As well, we can add a .rarity(Rarity.RARE) clause to make its name fancier. Lastly, we need to register our item. We DON’T need to register our armor material, as it’s just a variable that our new items use for their attributes. This new registry looks just like the ones we did before, swapping out the old <ITEM_NAME> and <item_name> for our new ones:

Registry.register(Registry.ITEM, new Identifier(main.MOD_NAME, "<item_name>"), <ITEM_NAME>);

We have successfully declared and registered our helmet! However, It has no item texture, and when you wear it, it has no model. Let’s change that.

Tags

A tag is essentially a list, or a set, of things in the game. Tags are divided by items, blocks, sounds, and game events. What tags are useful for is when you want to apply some function or attribute to multiple things at once, so rather than having to list every single item every time you want to use them, you can throw them once into a tag and then reference the tag instead. Let’s make it so Piglins are attracted to and will pick up Gilded Netherite equipment. If you did the additional challenge in the previous workshop, this will be very easy. You just have to add our new item to the list. If not, let’s make the tag JSON file fresh. Go to resources > data and make a directory called “minecraft”, then a directory called “tags”, then a directory in that called “items”. resources > data > minecraft > tags > items Make a new file titled “piglin_loved.json”. This is the exact name and directory of the tag used by Mojang when determining what Piglins like and will pick up. Use this format for the tag:

{
  "replace": false,
  "values": [
    "<namespace>:<item_name>",
		"<namespace>:<item_name>",
		...
  ]
}

The “replace” clause tells us whether or not we are replacing the contents of the tag if there is one with this name already, and the ... means that we can keep adding as many items as we like. Be sure to delete that after you’re done editing the file. Keep “replace” to false, and then add our brilliant alloy and all of our gilded netherite items to the list. Their namespace is ‘workshop’, and their <item_name> will be exactly what you wrote in their registry listings in the quotations. After this, there should be 5 items in the tag. Now piglins will get distracted by Gilded Netherite items!

Mixins

A mixin is complicated. You target a source class and then make changes to it based on the methods already in it. Think of it like tricking Minecraft into running some of your code inside its own files, and it thinks it belongs there and runs it like normal. There is a class file that determines what items the player can be wearing so that Piglins are not aggressive towards them. Currently, that is just gold armor, but we can add our Gilded Netherite armor to that list. Here’s how. Firstly, go to our workshop.mixins.json file under ‘resources’ and add “PiglinBrainMixin” to the “mixins” array (Include the quotation marks around it). Now, this file is trying to implement a mixin called exactly that, but there’s no file yet. Let’s make it. Go to java > net > workshop and first make a new package called mixin. Inside of that package, make a new class called PiglinBrainMixin. It is traditional to name your mixin class after the source class you are mixing into. The format of mixins is very complex and variable depending on the source class, so I think it’s best to show you the outline of our specific mixin, rather than a generic template:

@Mixin(PiglinBrain.class)
public class PiglinBrainMixin {
    @Inject(method = "wearsGoldArmor(Lnet/minecraft/entity/LivingEntity;)Z", at = @At(value = "INVOKE_ASSIGN", target = "Lnet/minecraft/item/ItemStack;getItem()Lnet/minecraft/item/Item;"), cancellable = true, locals = LocalCapture.CAPTURE_FAILHARD)
    private static void wearsGoldArmor(LivingEntity entity, CallbackInfoReturnable<Boolean> cir, Iterable iterable, Iterator var2, ItemStack itemStack) {
        
        }
    }
}

Let’s break this down. First, @Mixin tells us this is a mixin, and the PiglinBrain.class inside the parentheses is the class we are mixing into. After which is our own public class declaration, and then the body of it. We are going to do an @Inject, which means we are injecting new code into a pre-existing method. That method is wearsGoldArmor, which is the method behind determining if Piglins are aggressive towards players based on if they’re wearing gold armor or not. The next piece in parentheses is the path and return value of the call made to that method. The @At section tells us what kind of Inject we are doing with the value, in this case it’s an INVOKE_ASSIGN, which means our mixin allows the source class to call another method before it runs. That method is ItemStack.getItem(), referenced in the target section. Cancellable means that if something goes wrong, we can ignore this piece of code, we don’t have to run it. Lastly, locals will capture any error made while trying to run this code and send it to the terminal for us. This is good for debugging. The public static void WearsGoldArmor() piece is the setup for the actual method, and the parameters inside are exactly what the source method uses for its body. We do not need to change anything here. Now let’s add to the body of the method. First, we need to grab the ID of whatever we have equipped. We can do that by making an Item and declaring it to be a call to the itemStack.getItem() method. It looks like this:

Item item = itemStack.getItem();

I just called it item because that’s short and easy, but it’s probably not a great name. Not very descriptive. something like gildedItem would be better. Now let’s do the logic. Below our item/gildedItem, make an if statement. Inside the parentheses, we need to check two things:

  1. Is an item we have equipped armor?
    1. first, place item in the parentheses. we will use a special comparison called instanceof, which checks the object type of the item. To finish this comparison, we will use the object type we want, which is ArmorItem. The first part of the comparison is: item instanceof ArmorItem
  1. Is that armor piece Gilded Netherite?
    1. Put a && after the first section so we have to check both pieces of our if statement for it to work. Start some parentheses with a ( symbol. Next, we need to bring in our armor object. The way you do this in Java is you put the object type in parentheses, then the name of the object. Then, close the parentheses. Next, now that we’re using an ArmorItem, we can make a call to its .getMaterial() method. You should have this: ((ArmorItem) item) Now we can compare. Add a == symbol after to compare values. We are going to compare it to our armor variable that we made in ItemInit, the one with <MATERIAL_NAME>. So, the second clause of the statement is: ((ArmorItem) item).getMaterial() == ItemInit.GILDED_NETHERITE_ARMOR

Inside the body of the if statement, we only need to return a true, as in the player is wearing “gold” and so Piglins will not attack them. The body only needs this line: cir.setReturnValue(true); In total, it looks like: 

@Mixin(PiglinBrain.class)
public class PiglinBrainMixin {
    @Inject(method = "wearsGoldArmor(Lnet/minecraft/entity/LivingEntity;)Z", at = @At(value = "INVOKE_ASSIGN", target = "Lnet/minecraft/item/ItemStack;getItem()Lnet/minecraft/item/Item;"), cancellable = true, locals = LocalCapture.CAPTURE_FAILHARD)
    private static void wearsGoldArmor(LivingEntity entity, CallbackInfoReturnable<Boolean> cir, Iterable iterable, Iterator var2, ItemStack itemStack) {
        Item item = itemStack.getItem();
        if (item instanceof ArmorItem && ((ArmorItem) item).getMaterial() == ItemInit.GILDED_NETHERITE_ARMOR) {
            cir.setReturnValue(true);
        }
    }
}

And we’re done! Piglins will no longer attack players wearing gilded netherite armor.

Knockback Resistance Mixin

We can actually make our value for knockback resistance work with a mixin. For some odd reason, that attribute and ONLY that attribute is hard-coded by Mojang, specifically so that only netherite is the only armor that gets it. Our work-around will be to use a mixin to force the game to give our new armor the attribute. Go to java > net > workshop > mixin and make a new class called ArmorItemMixin. Make sure to add this name to our workshop.mixins.json file in resources. This is a mixin, so add @Mixin(ArmorItem.class) Before our public class declaration. Then, in our public class, add the word “abstract” between the words so that is now reads public abstract class ArmorItemMixin { Now here’s a very fun part. There is a LOT of complicated Java stuff happening in this mixin, stuff that even I’m not fully sure on the details of, but I will do my best to explain what is happening. The first section @Shadows some variables from the superclass, as these three lines: 

@Shadow @Final private static UUID[]MODIFIERS;
@Shadow @Final @Mutable private Multimap<EntityAttribute, EntityAttributeModifier> attributeModifiers;
@Shadow @Final protected float knockbackResistance;

The MODIFIERS array is the array of the values we placed into our ArmorGildedNetherite class. The Multimap is like a graph of those modifiers, so the system can keep track of them. The protected float knockbackResistance is that value that controls that attribute, and the fact that it’s protected is the reason we have to do this in the first place. The next part is the @Inject. It goes like this:

@Inject(method = "<init>", at = @At(value = "RETURN"))
private void constructor(ArmorMaterial material, EquipmentSlot slot, Item.Settings settings, CallbackInfo ci) {
    UUID uUID = MODIFIERS[slot.getEntitySlotId()];
    if (material == ItemInit.GILDED_NETHERITE_ARMOR) {
        ImmutableMultimap.Builder<EntityAttribute, EntityAttributeModifier> builder = ImmutableMultimap.builder();

        this.attributeModifiers.forEach(builder::put);

        builder.put(
                EntityAttributes.GENERIC_KNOCKBACK_RESISTANCE,
                new EntityAttributeModifier(uUID,
                        "Armor knockback resistance",
                        this.knockbackResistance,
                        EntityAttributeModifier.Operation.ADDITION
)
        );

        this.attributeModifiers = builder.build();
    }
}

Firstly, the @Inject’s method target the “init”, meaning it runs this code on startup, and value RETURN adds this code to the return section of the superclass. This means that this mixin will first run when the class is loaded, and immediately return whatever values it gets. We then declare a universally unique identifier (UUID) for each set of MODIFIER values found on startup. As in, for each armor material we find. If the material superclass of that modifier is ArmorGildedNetherite (ItemInit.GILDED_NETHERITE_ARMOR), then it builds a new Multimap for every piece of armor it finds with that superclass (all four armor pieces we made). It then does a forEach loop, where it will run builder.put for each piece of armor in the Multimap. Inside of builder.put, it grabs the value for GENERIC_KNOCKBACK_RESISTANCE (currently 0), and then it grabs a new attribute, the value for knockback resistance that we put into ArmorGildedNetherite, and gives it a name (which is what’s in the quotations). It finally adds those two values together, and then the constructor builds the armor piece’s new attributes, which means it now has knockback resistance.
